> **Purpose**: To catch regression errors *immediately* after modification.

**The Rule**:
> NEVER edit a file without verifying the build/lint state immediately after.

**Execution**:

1. **Edit** the file.
2. **IMMEDIATELY** run a validation command (`npm test`, `python3 -m py_compile`, `eslint`, or `get_errors` if available).
3. **If Error**: Auto-correct if trivial, or report to user if complex.
4. **Log**: Include validation status in the final response.

**Why**: An edit is not "done" until it compiles.

### 0.8 Continuous Improvement Protocol (Recurring Issue Auto-Fix)

**Philosophy**: When the same friction appears twice, it becomes a workflow bug. Fix it autonomically.

**Trigger → Action Matrix**:

| Pattern Detected | Auto-Action | Example |
|------------------|-------------|---------|
| User manually does X twice | Automate X into workflow | "Auto-checkpoint instead of /save" |
| Same error/blocker recurs | Create prevention protocol or script | "Node.js missing → default to Static HTML" |
| User expresses frustration at repetition | Document fix in relevant workflow | "Log this into your workflow" |
| Workaround discovered | Formalize as capability | "Netlify Drop for instant hosting" |

**Tags**: #workflow #automation #vibe

**Execution Protocol**:

1. Detect pattern (2+ occurrences of same friction).
2. Propose fix (workflow edit, new script, new protocol).
3. Execute fix (no permission for non-destructive workflow edits).
4. One-liner confirmation: "Fixed. [X] now happens automatically."

**Autonomic Behavior**: This runs implicitly. User sees outcome, not process.
